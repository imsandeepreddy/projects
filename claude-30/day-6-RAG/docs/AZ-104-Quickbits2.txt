- In a storage account, if you want one of the blobs to use different key for encryption first "Create an encryption scope" and then assign this scope to specific blob.
- azcopy copy D:\folder1 https://contosodata.blob.core.windows.net/public --recursive is the correct answer because the azcopy copy command is used to upload files from an on-premises folder to an Azure Storage container. The --recursive option ensures that all files within D:\Folder1 are copied to the destination container in Azure.
- The net use command relies on the SMB (Server Message Block) protocol, which only supports authentication using the Azure Storage account key for access to Azure File Shares. The SAS token cannot be used for authentication in this context, and attempting to use it will result in an error indicating invalid credentials.
- Azure Key Vault supports only RSA keys for customer-managed storage account encryption and max bit length is 4096.
- SMB (Server Message Block) protocol, which is used to access Azure file shares from Windows client, operates over port 445. Windows clients need access to port 445 to map a drive to the Azure file share.
- Azure file provides a fully managed SMB file share that can be mounted by Azure container instances as persistent storage. This is essential for scenarios like running Microsoft SQL Server in a container, where data needs to persist across container restarts. However, it is important to note that volume mounting, including Azure Files, is currently only supported for Linux-based Azure container instances. 
- StorageV2 with LRS can be converted into ZRS but not BlobStorage(not eligible for ZRS replication)
- StorageV2 with RA-GRS can be converted into ZRS by first changing it to LRS and then to ZRS.
- Signing key refers to storage account access key, not a SAS token. Storage account keys act like root-level credentials and grant unrestricted full access to all services and data types in the storage account
- StorageV2, BlobStorage, and BlockBlobStorage—support lifecycle management policies for block blobs and append blobs.
- Only storage accounts configured with LRS (Locally Redundant Storage), GRS (Geo-Redundant Storage), or RA-GRS (Read-Access Geo-Redundant Storage) support moving blob data to the Archive access tier. if any has zone, it wont support archive.
- AzCopy supports copying data to Blob storage and File storage, but not Table storage or Queue storage. AzCopy is specifically designed for transferring blob and file data efficiently within and across Azure storage accounts.
- Role assignment conditions in Azure can enforce granular access control based on blob index tags.
- Hot: No minimum retention period; early deletion penalty does not apply.
- Cool: 30 days minimum retention period.
- Cold: 90 days minimum retention period.
- Archive: 180 days minimum retention period.
