Backup:
- Azure file share in storage1 is eligible for backup by the Recovery Services vault. Blobs are not supported for backup in a Recovery Services vault.
- Restore VM1 by using the Create new restore configuration option is correct because it allows you to recover VM1 to a previous state from eight days ago without affecting the existing VM. This approach minimizes downtime as it creates a new VM instance based on the backup from eight days ago, enabling you to test and verify the restored version separately before making it live if needed.
- VM1 is a Windows Server virtual machine that uses Azure Disk Encryption and is intended to be backed up using a Recovery Services vault. According to Azure backup support policies, Standard policy types in Recovery Services vaults fully support backing up Windows VMs with Azure Disk Encryption enabled. 
- Azure supports backing up Linux virtual machines with Trusted Launch when using Enhanced policies in Recovery Services vaults. Enhanced policies are specifically designed to support advanced VM security configurations like Trusted Launch, and they offer features such as backup of selected disks and multiple backups per day.
- Multi-User Authorization (MAU) in a Recovery Services vault requires creating a resource guard.
- Azure blob container/Azure Disks can only be backed up in Azure Backup.
- Azure files, VMs can only be backed up to Azure Recovery Services Vault.
- Allow Azure services on the trusted services list to access this storage account and enable it - For Azure Backup to backup disks here.
- StandardV2 supports blob-level tiering. So individual blob can have different tiers.
- Simply having the role of Global Admin does not allow the user access to file share data in storage account.
-  Azure Storage supports customer-managed keys (CMK) for encryption using RSA keys only, and those keys must have a key size of 2048, 3072, or 4096 bits.

Storage:
- daysAfterModificationGreaterThan specifies the condition to move blobs that have not been modified within the specified period (45 days, in this case) to the Cool access tier.
- Encryption scopes in Azure Storage are applicable to Azure Blob Storage. Since blobs are stored within containers, encryption scopes can be used to encrypt containers, but not other storage types such as file shares, tables, or queues.
- General Purpose v2 storage, which supports both Azure file shares and tiering.
- StorageV2 (general purpose v2) offers the most versatile functionality, supporting blobs, files, queues, and tables. This storage type is the recommended choice when planned changes may require flexibility in storage capabilities, as it supports all three access tiers (Hot, Cool, and Archive) and integrates with advanced features, such as lifecycle management and storage tiering.
- SMB Multichannel is only supported on Premium File Shares, which use SSD-based storage.
- To minimize the network costs of accessing storage account, modify the default routing tier setting.
- After storage account creation, you can modify the Encryption type setting.
- Allow storage account key access is disabled, meaning that any SAS signed with the account keys (such as an Account SAS or a Service SAS) cannot be used. However, a User Delegation SAS is signed with a user delegation key that is derived from the user’s Microsoft Entra ID credentials, not from the storage account keys.
